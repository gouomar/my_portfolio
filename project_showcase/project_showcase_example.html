Agency Brain

A multilingual WhatsApp assistant whose routing once rested on two brittle heuristics — a keyword regex and a single similarity score. I replaced both with constrained-decoded classifiers that always return a clean, structured decision, and kept Gemini for the one thing it's irreplaceable at: writing the answer.

RAGconstrained decodingFastAPIPython

Students text in French, Arabic, and Darija, at all hours, and expect a human-quality answer.

The interesting work here isn't the model. It's the orchestration — knowing which of many moving pieces should make each decision, and wiring them so the whole thing behaves predictably even when one piece misbehaves. This page walks that story from the plainest framing down to the measured numbers.

the problem

One dumb pipe doesn't survive contact with production

A study-abroad agency in Morocco lives inside its WhatsApp inbox. Students fire off questions all day, in three languages and one romanized street dialect, braided into a single thread. Every unanswered message at 11pm is a lost client.

The naive build is one dumb pipe: forward every message to an LLM, return whatever comes back. It works in a demo and falls apart in production — you pay for a model call on "salam", and you pay again when someone asks about the weather.

The real problem here was never generation. It was orchestration — deciding, cheaply and reliably, which messages deserve an expensive answer before a single dirham is spent.

And here's the part most people get backwards: in the original system, the expensive model never made those decisions at all. Routing rested on two cheap heuristics — a keyword regex to spot greetings, and a single cosine-similarity number to guess whether a question was in scope. Those two heuristics were the fragile part. This project replaces them with something trustworthy.

in plain terms

A few words, used throughout

knowledge base
the agency's own documents — visa rules, tuition, university lists, deadlines. The assistant may answer only from these. It never invents facts.
in / out of scope
in scope if the KB actually contains the answer. Out of scope if it doesn't — those messages get handed to a human instead of guessed at.
routing
deciding which lane a message belongs in. Most of this case study is the story of making that one decision reliable.
constrained decoding
forcing a model's output to fit a fixed shape — a label, or a typed schema — at generation time, so it is literally incapable of returning anything unusable.

how it works

Two small classifiers, not one big guess

The routing isn't one decision — it's two constrained classifiers placed at the forks that used to be guesswork, with a cheap embedding-score shortcut in front of the second so it only runs when actually needed.

the only box that costs money is the amber one

The intent gate runs first on the raw message. It doesn't pattern-match for greetings — it decides, in one constrained yes/no, whether a real question hides underneath. A greeting glued to a question, like "salam, les universités au Canada?", is never swallowed as smalltalk the way a regex would swallow it.

The scope gate is deliberately not one decision. Running a 1.5B model on every message is slow — around seven seconds on CPU — and most messages don't need it, since the retrieval score already screams the answer. Only the ambiguous middle band, [0.60, 0.68), actually pays for the answerability classifier, which reads the question and the retrieved chunk together.

Spend the expensive check only where the cheap signal is uncertain — that's the whole design philosophy in one gate.

three decisions shaped this system, each chosen over the obvious alternative

Two classifiers, not two heuristics

A general-purpose LLM asked "is this in scope?" will eventually answer "it depends" and break the router. Small model, locked output, guaranteed-parseable decision.

Retrieve-first, not query-in-one-shot

Splitting retrieve() from generate() means the scope check can read actual evidence before anyone spends on generation. Evidence before spend.

A dumb bridge, not a smart one

The bridge only relays text and a session id. All intelligence lives in one runtime — one place to reason about, one place to upgrade.

the hard parts

Four problems that had to be solved on purpose

You can't pipe a trilingual WhatsApp inbox into an LLM and hope.

The Darija wall

A keyword regex catches salam and misses slm. It can't tell a greeting glued to a real question from a plain hello, so the question silently dies in the wrong lane.

the fix

The intent classifier reads the whole message, dialect and all, and emits a constrained yes/no on whether a request hides underneath. The greeting can never hide the question.

The follow-up paradox

A student asks about Canada, gets an answer, then sends "chhal taman?" — "how much?". On its own that fragment scores low and a strict scope gate would reject a valid follow-up.

the fix

A memory buffer keyed to the student's phone number lets the scope gate trust context outright when a session is active, resolving the fragment against the thread it remembers.

One number can't carry a decision

The old in-scope check was a single threshold: clear 0.66, answer; otherwise hand off. Two real, answerable Arabic questions score under it — and out-of-scope noise sits in the same band, so lowering the cutoff would let junk through too.

the fix

A classifier reads the question and the retrieved chunk together and judges whether the evidence actually supports the answer — semantic judgment exactly where a raw score chokes, the cheap threshold everywhere else.

Telling the truth when the brain is offline

Two runtimes, one machine — WhatsApp in Node, intelligence in Python, talking over localhost. If Python is down, a naive bridge hangs the chat and the student assumes they've been ignored.

the fix

Every call to the brain has a timeout. A refused connection triggers a graceful, human-toned fallback instead of a frozen chat. The dumb pipe's one job is honesty.

The real value of constrained decoding

On binary yes/no gates, constrained decoding shows no measurable advantage — a 1.5B model asked "yes or no" already answers cleanly. The delta appears the instant the decision has structure, tagging a query as a typed {country, topic} object.

the honest thesis

Constrained decoding's payoff isn't accuracy on trivial gates — it's that the moment your output has structure, it deletes an entire class of parse-and-validate bugs by construction. On a binary flag that's a nice guarantee; on a schema it's the difference between 39% and 100% usable.

Two concrete failure modes the constrained path makes impossible: 61% of free outputs wrapped the JSON in markdown fences, and even a forgiving parser still failed on out-of-enum values it couldn't repair — "Canada" in the wrong case, "Morocco" not a valid option at all.

what I'd do differently

The honest punch list

  • Make the decision schemas pluggable. The {country, topic} tagger already proves the schema path works — declare all decisions as named schemas and add new ones without touching generation.
  • Fold the follow-up rule into the classifier. The "active session → trust context" bypass is still a hand-written Python branch. Feed conversation-state straight into the answerability classifier instead.
  • Get the scope-gate latency down. ~7s per gray-band decision is fine for WhatsApp but not great. A quantized model or a smaller distilled judge would cut it.
  • Promote logging to a real store. A JSONL file is fine for one machine; an agency running this for months wants SQLite or Airtable for actual lead analytics.
